Wat is REST?

Softwarearchitectuur

Volgens de definitie van Fielding is een software architectuur “een abstractie van de run-time elementen van een software systeem tijdens een fase van operatie.” De elementen zijn onderverdeeld in drie verschillende typen: componenten, data en connectors. We werken deze verschillende typen kort uit.

Components

[Fielding 2000]

De componenten van een software architectuur zijn elementen en instructies die data kunnen veranderen, die aangestuurd wordt door zijn eigen interface. Om dit concreet te maken een lijst met voorbeelden wat de functie van een component is:

  • Berekeningen uitvoeren
  • Veranderen van een formaat
  • Samenvoegen met andere data
  • Genereren van data

Connectors

De functie van connectors is simpel: het verbinden van verschillende componenten met elkaar. Het kan wel veranderingen doorvoeren op de data, maar niet op het niveau van de echte data. Zo kan het wel intern de data transformeren met het doel om de data beschikbaar en bruikbaar te maken voor de ontvanger. Zo is bijvoorbeeld het HTTP protocol een voorbeeld van een connector.

Data

Data is wat verstuurd of ontvangen wordt door een component via een connector.

Implementatie REST

Components

Connectors

In REST zijn er vijf connectors: client, server, cache, resolver en tunnel. Alle vijf zijn ze stateless - dat is immers de filosofie achter REST. De client doet requests naar de server, de server wacht op requests van de client. Het cache kan op de interface van én de client én de server zitten. Het is mogelijk om een groot cache extern op te bouwen, maar het gevaar hiervan is dat er snel asynchronisatie op kan treden.

Data

De data-elementen zijn ruwweg onderverdeeld in twee categorieën: resources en representations. In die twee categorieën zijn er weer verdelingen.

Resources

Resources zijn de belangrijkste data-elementen van REST. Een resource is erg abstract, dat wil zeggen dat een resource een document, afbeelding of ‘het weer in Utrecht’. Het is dus heel abstract en gaat niet in op de echte syntax van de data, meer om de semantiek. Resources kunnen ruwweg in twee categorieën verdeeld worden, namelijk statische en dynamische resources. Een resource kan opgehaald worden door middel van een resource identifier.

Representations

Representations zijn bytes, ofwel de data zelf. Dit kan bijvoorbeeld een HTTP entity of een bestand zijn. Ook is de metadata over de representation aanwezig, zoals de tijd van aanpassen van de file, bestandsformaat of mediatype. Daarnaast is er nog de Control Data die extra parameters kan geven aan de resources.

Stapsgewijze opbouw REST architectuur

1.Null style ↓

Dit is het begin en uitgangspunt zonder enige restricties en regels. Hier worden langzaam maar zeker regels aan toegevoegd.

2.Client-server ↓

De eerste constraint is die van het client-server model. Dit gaat uit van het principe dat de user interface afgesplitst is naar de client, zodat de server de focus kan leggen op het serveren van de data en de applicatie mobieler wordt. Een ander voordeel is dat beide kanten onafhankelijk verbeterd kunnen worden.

3.Stateless ↓

Deze constraint zorgt ervoor dat de client de state zelf op moet bouwen, zodat de server hier geen informatie over hoeft op te slaan. Als de server hier geen informatie voor hoeft op te slaan is het mogelijk om requests onafhankelijk af te handelen wat voor minder interferentie zorgt, wat zorgt voor een betere schaalbaarheid. Toch heeft dit weer nadelen, zoals de mogelijke toename van netwerkgebruik. Aangezien de server de state niet onthoudt moet de client telkens de state meesturen als deze nodig is voor de server om de request af te handelen. Ook is het moeilijker voor de server om voor een consistente applicatie te zorgen aangezien de server hier geen controle over heeft.

4.Cache ↓

Door het toevoegen van een cache is het mogelijk om de latency naar de server te verkleinen en minder resources nodig te moeten hebben om een request af te handelen. De server kan bepalen wat cacheable en niet-cacheable is, maar in veel gevallen moet de client zelf optimalisering doorvoeren, aangezien de server maar een beperkte controle hierover heeft. Het is namelijk mogelijk dat de server zegt dat een document cacheable is maar de client hier niets mee doet en constant een nieuwe request uitvoeren.

5.Uniforme Interface ↓

Een uniforme interface zorgt ervoor dat het hele systeem versimpeld kan worden aangezien er geen extra gespecialiseerde versies voor bepaalde clients is. Dit heeft tot gevolg dat het overzichtelijker wordt. De REST interface is dus gespecialiseerd in het uitserveren van data naar grote hoeveelheden clients. De onafhankelijkheid van client en server zoals genoemd in punt twee wordt nu versterkt - aangezien de interface uniform is, is het mogelijk om aan beide kanten veranderingen en optimaliseringen door te voeren zonder dat dit direct de communicatie van beide componenten beïnvloedt. De nadelen van dit systeem is dat de efficiëntie minder wordt omdat er een gestandaardiseerde interface komt die mogelijk te veel of te weinig informatie aanlevert wat niet in het optimale formaat is voor de client.

6.Layered System ↓

Een layered system betekent dat de componenten in het systeem layered zijn, wat betekent dat de componenten op verschillende lagen geen weet hebben van de andere lagen. Dit kan gebruikt worden om nieuwe en oude systemen werkend te krijgen onder een enkele uniforme interface zoals beschreven in punt vijf. Het grootste nadeel hiervan is er een overhead komt en de performance voor de gebruiker vermindert.

7.Code on Demand ↓

Deze eigenschap zorgt ervoor dat een client code van de service kan halen om zelf uit te voeren. Zo is de server wel meer in controle - bijvoorbeeld om features dynamisch toe te kunnen voegen - maar wordt de service wel weer onduidelijker omdat de scheiding tussen client en server begint te vervagen. Daarom is deze eigenschap ook optioneel en is het niet ongebruikelijk om deze achterwege te laten.

8.Samenvatting

Al deze eigenschappen bij elkaar gevoegd creëren een systeem dat bijzonder schaalbaar is en de nodige voor- en nadelen afweegt. De grootste voordelen op een rijtje:

  • Portability
  • Extendibility
  • Scalability
  • Independency

Wat is SOAP?

SOAP staat voor Simple Object Acces Protocol, en is na REST de populairste manier om met webservices te communiceren. SOAP is ontworpen door Microsoft in 1998. Maar werd door verschillende redenen binnen Microsoft pas in 1999 uitgegeven. In 2000 werd SOAP 1.1 uitgegeven door het World Wide Web Consortium (W3C). Tegenwoording wordt SOAP onderhouden en verder ontwikkeld door de XML Protocol Working Group van W3C. SOAP is ontworpen om erg uitbreidbaar te zijn. Men hoeft alleen maar de delen van SOAP te gebruiken die zij nodig hebben voor bepaalde taken. SOAP heeft bijvoorbeeld de optie voor extra beveiliging, maar die hoeft niet gebruikt te worden als het niet nodig is, bijvoorbeeld bij een message voor een simpele openbare tekstapplicatie. Echter zou deze extra beveiliging wel bijzonder handig zijn als de gegevens die worden verstuurd van een bank zijn. Een belangrijk onderdeel van SOAP is het ingebouwde fout afhandeling, als er een fout zou zijn in je requestmessage wordt in de response message een fout code en tekst toegevoegd waardoor je direct weet wat er aan de hand is. Doordat er standaard foutcodes worden gebruikt kan je in de code van je programma ook direct de fout laten afhandelen. Een andere interessante mogelijkheid van SOAP is dat je niet verplicht http als transport protocol hoef te gebruiken. Met SOAP kan je ook ander transport protocollen gebruiken, zoals SMTP of andere transport methodes.

In het kort samengevat zijn de drie belangrijkste eigenschappen van SOAP:

  • Uitbreidbaarheid

    SOAP is heeft heel erg veel mogelijkheden om uitgebreid te worden, bijvoorbeeld er kan extra beveiliging worden toegevoegd.

  • Neutraliteit

    SOAP is niet alleen mogelijk op met één transport protocol, maar kan gebruikt worden met verschillende transport protocollen, zoals http en SMTP.

  • Onafhankelijkheid

    SOAP kan gebruikt worden met verschillende programmeertalen, zoals PHP en .NET.

Waaruit bestaat SOAP?

Het SOAP protocol bestaat uit drie delen:

  • Envelope

    Een SOAP bericht bestaat uit een envelope met daarin optioneel een header en altijd een body. De envelope moet altijd geassocieerd zijn met de namespace "http://schemas.xmlsoap.org/soap/envelope/". Is dit niet het geval dan word de message weggegooid.

    • Header

      De Header wordt gebruikt als een flexibel mechanisme voor het uitbreiden van een bericht zonder voorafgaande kennis tussen de communicerende partijen. In de header kan bijvoorbeeld transactie management of betaling worden geïmplementeerd.

    • Body

      In de body staat informatie die voor de uiteindelijke ontvanger is bestemd, in de body staat dus de data die bijvoorbeeld is opgevraagd door iemand. Verder kan er in de body als er een fout is geweest ook nog wat extra informatie staan over deze fout, zoals de foutcode, en een leesbare beschrijving van de fout.

  • Encoding Rules

    Een SOAP message is in XML geschreven echter zijn er wel nog meer beperkingen opgelegd aan hoe het geschreven mag worden.

  • RPC Representation

    Een belangrijk doel van SOAP is om Remote Procedure Calls uit te wisselen door gebruik van de flexibiliteit van XML, normaal gesproken worden RPC’s gedaan met een HTTP Request en HTTP Response, maar met SOAP hoeft dit niet via http, omdat SOAP ook van andere transfer protocols gebruik kan maken zoals het Simple Mail Transfer Protocol.

Verwerking van SOAP

Als een SOAP applicatie een SOAP message ontvangt, wordt dit bericht dit bericht geprocesseerd worden dit gebeurd in een bepaalde volgorde:

1. Identificatie ↓

Als eerste moet de applicatie alle delen van de SOAP message identificeren die voor de applicatie bedoeld zijn, het zou namelijk kunnen dat de SOAP message nog verder moet worden gestuurd.

2. Verificatie ↓

Hierna moet de applicatie verifiëren dat al deze delen worden ondersteund door de applicatie, is dit het geval dan kunnen deze op de juiste manier worden geprocesseerd. Is dit echter niet het geval en ondersteund de applicatie de delen niet, dan word de message weggegooid.

3. Afhandeling

Deze laatste stap gebeurt alleen als deze applicatie niet de eindbestemming is van de SOAP message. Als de SOAP message nog verder moet gaan worden alle delen verwijdert die bij stap 1 waren geidentificeerd, omdat deze alleen voor deze applicatie waren bestemd. Hierna kan de message geforward worden.

Voor- en nadelen van SOAP

Voordelen:

  • Het is te gebruiken op meerdere platforms en kan ook gebruikt worden met verschillende programmeer talen.
  • SOAP geeft makkelijkere communicatie door proxies en firewalls.
  • Het kan gebruikt worden met verschillende transportprotocollen, waarvan de bekendste http en SMTP zijn, maar het kan ook met andere transportprotocollen gebruikt worden.

Nadelen:

  • SOAP is over het algemeen langzamer concurrenten, omdat SOAP gebruikt maakt van XML waarin meer informatie is toegevoegd dan echt nodig is. Hierdoor zijn de messages groter en duurt het dus langer om deze te versturen en te verwerken. Dit is een belangrijke reden waardoor tegenwoordig veel meer mensen gebruik maken van REST
  • SOAP wordt niet door alle programmeertalen even sterk ondersteund. Python en PHP worden bijvoorbeeld niet even goed ondersteund als Java en .NET.

SOAP vs. REST

SOAP is heavyweight, en voor veel mensen lastig te gebruiken.

  • SOAP kan meerdere transportprotocolllen gebruiken zoals http en SMTP, REST alleen http
  • Makkelijk uit te breiden
  • Built-in error handling zodat wanneer er een error is de ontvanger direct een idee heeft wat het probleem is.

REST is lightweight, en makkelijk te gebruiken en is flexibeler.

  • Makkelijker te leren
  • Efficiënter, kan verschillende dataformaten gebruiken zoals JSON waardoor het bestand kleiner is. SOAP kan alleen XML gebruiken waar allerlei toegevoegde informatie zit die niet relevant is voor de ontvanger.
  • Snel, geen uitgebreide verwerking nodig, maar toch de mogelijkheid voor grootschaligheid
