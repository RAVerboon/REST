Wat is REST?

Softwarearchitectuur

Volgens de definitie van Fielding is een software architectuur “een abstractie van de run-time elementen van een software systeem tijdens een fase van operatie.” De elementen zijn onderverdeeld in drie verschillende typen: componenten, data en connectors. We werken deze verschillende typen kort uit.

Components

De componenten van een software architectuur zijn elementen en instructies die data kunnen veranderen, die aangestuurd wordt door zijn eigen interface. Om dit concreet te maken een lijst met voorbeelden wat de functie van een component is:

  • Berekeningen uitvoeren
  • Veranderen van een formaat
  • Samenvoegen met andere data
  • Genereren van data

Connectors

De functie van connectors is simpel: het verbinden van verschillende componenten met elkaar. Het kan wel veranderingen doorvoeren op de data, maar niet op het niveau van de echte data. Zo kan het wel intern de data transformeren met het doel om de data beschikbaar en bruikbaar te maken voor de ontvanger. Zo is bijvoorbeeld het HTTP protocol een voorbeeld van een connector.

Data

Data is wat verstuurd of ontvangen wordt door een component via een connector.

Architecturale elementen

REpresentational State Transfer stijl is een abstractie van architecturale elementen in een gedistribueerd systeem. Perry en Wolf hebben drie klassen gedefinieerd voor architecturale elementen: componenten, data elementen en connectors. De REST stijl legt de focus op op de rol die de componenten spelen in plaats van de implementatie daarvan.

Stapsgewijze opbouw REST architectuur

1.Null style ↓

Dit is het begin en uitgangspunt zonder enige restricties en regels. Hier worden langzaam maar zeker regels aan toegevoegd.

2.Client-server ↓

De eerste constraint is die van het client-server model. Dit gaat uit van het principe dat de user interface afgesplitst is naar de client, zodat de server de focus kan leggen op het serveren van de data en de applicatie mobieler wordt. Een ander voordeel is dat beide kanten onafhankelijk verbeterd kunnen worden.

3.Stateless ↓

Deze constraint zorgt ervoor dat de client de state zelf op moet bouwen, zodat de server hier geen informatie over hoeft op te slaan. Als de server hier geen informatie voor hoeft op te slaan is het mogelijk om requests onafhankelijk af te handelen wat voor minder interferentie zorgt, wat zorgt voor een betere schaalbaarheid. Toch heeft dit weer nadelen, zoals de mogelijke toename van netwerkgebruik. Aangezien de server de state niet onthoudt moet de client telkens de state meesturen als deze nodig is voor de server om de request af te handelen. Ook is het moeilijker voor de server om voor een consistente applicatie te zorgen aangezien de server hier geen controle over heeft.

4.Cache ↓

Door het toevoegen van een cache is het mogelijk om de latency naar de server te verkleinen en minder resources nodig te moeten hebben om een request af te handelen. De server kan bepalen wat cacheable en niet-cacheable is, maar in veel gevallen moet de client zelf optimalisering doorvoeren, aangezien de server maar een beperkte controle hierover heeft. Het is namelijk mogelijk dat de server zegt dat een document cacheable is maar de client hier niets mee doet en constant een nieuwe request uitvoeren.

5.Uniforme Interface ↓

Een uniforme interface zorgt ervoor dat het hele systeem versimpeld kan worden aangezien er geen extra gespecialiseerde versies voor bepaalde clients is. Dit heeft tot gevolg dat het overzichtelijker wordt. De REST interface is dus gespecialiseerd in het uitserveren van data naar grote hoeveelheden clients. De onafhankelijkheid van client en server zoals genoemd in punt twee wordt nu versterkt - aangezien de interface uniform is, is het mogelijk om aan beide kanten veranderingen en optimaliseringen door te voeren zonder dat dit direct de communicatie van beide componenten beïnvloedt. De nadelen van dit systeem is dat de efficiëntie minder wordt omdat er een gestandaardiseerde interface komt die mogelijk te veel of te weinig informatie aanlevert wat niet in het optimale formaat is voor de client.

6.Layered System ↓

Een layered system betekent dat de componenten in het systeem layered zijn, wat betekent dat de componenten op verschillende lagen geen weet hebben van de andere lagen. Dit kan gebruikt worden om nieuwe en oude systemen werkend te krijgen onder een enkele uniforme interface zoals beschreven in punt vijf. Het grootste nadeel hiervan is er een overhead komt en de performance voor de gebruiker vermindert.

7.Code on Demand ↓

Deze eigenschap zorgt ervoor dat een client code van de service kan halen om zelf uit te voeren. Zo is de server wel meer in controle - bijvoorbeeld om features dynamisch toe te kunnen voegen - maar wordt de service wel weer onduidelijker omdat de scheiding tussen client en server begint te vervagen. Daarom is deze eigenschap ook optioneel en is het niet ongebruikelijk om deze achterwege te laten.

8.Samenvatting

Al deze eigenschappen bij elkaar gevoegd creëren een systeem dat bijzonder schaalbaar is en de nodige voor- en nadelen afweegt. De grootste voordelen op een rijtje:

  • Portability
  • Extendibility
  • Scalability
  • Independency
